## Walkthrough

We are given a password-protected zip file and a metadata.txt file, which contained the password to the zip. However, I was dumb and attempted to brute force the password using rockyou.txt for the first 20 minutes, before reading the metadata file and finding the password inside :skull:

Unzipping the file, we are given an ad1 file, which was a (proprietary?) type of logical image developed by AccessData, and could be opened using FTK Imager. By opening the file in FTK Imager, we can peek into the file structure of the drive at the time of imaging.&#x20;

<figure><img src="../../.gitbook/assets/image (154).png" alt=""><figcaption><p>File structure of the image file</p></figcaption></figure>

At first glance, I noticed the csitfan1 user folder, which was not a default user. Looking through the contents of the folder, I noticed some .lnk files (linker/shortcut files) to a flag.txt file (this turned out to be a red herring, but I didn't realise at the time)

<figure><img src="../../.gitbook/assets/image (155).png" alt=""><figcaption><p>The first red herring, 30 minutes in</p></figcaption></figure>

Not knowing that there were red herrings at the time, I spent 15 minutes trying to restore the deleted file, before I stumbled on the restore points folder, stored in the System Volume Information folder

{% code overflow="wrap" %}
```
Info: Restore points are similar to save points in MMO games, allowing a local administrator to revert the system to a previous point in time. This can help with forensics and dynamic malware analysis efforts, as well as recovering from ransomware, assuming the restore points weren't encrypted or destroyed
```
{% endcode %}

<figure><img src="../../.gitbook/assets/image (156).png" alt=""><figcaption><p>RP: Restore Points</p></figcaption></figure>

In RP2, there was a change.log file, which was automatically generated by the system and contains the list of files added, removed or modified (kind of like a git PR). However, this was also a red herring, as there wasn't any way to load RP2 into a WinXP VM (which was thankfully pretty easy to find)

I was lost at this point, so I just spent the next 8 hours alternating between level 3 TISC and my school trainings. I searched through every folder in the csitfan1 user directory, before switching to trying the default users and services directories (I was getting desperate). For some reason, I didn't notice the Mypal folder sitting in the csitfan1 user Desktop's folder, probably assuming that it was another red herring. At this point, I was getting quite desperate (and tired), so I decided to check what Mypal was.

Looking at the official page for the app, it seemed like Mypal is a fork (?) of Mozilla Firefox, designed for Windows XP (the OS of the machine that we're inspecting). The file structure for Mypal seemed to be similar to Firefox (no surprise, in hindsight), so I looked up the structure of a Firefox profile directory (official [link](https://support.mozilla.org/en-US/kb/profiles-where-firefox-stores-user-data)).

Straight off the bat, I see something of interest

<figure><img src="../../.gitbook/assets/image (157).png" alt=""><figcaption><p>Search history? Interesting</p></figcaption></figure>

In total, I marked out several files that I wanted to look for in the profile. These files were:

* places.sqlite - no-brainer, maybe the browsing history contains the flag, or a URL to the flag
* key4.db and logins.json - possible user credentials to access some secret website or page
* formhistory.sqlite - same reasoning as places.sqlite, maybe the user wrote the flag in a form somewhere
* cookies.sqlite3 - possible JWT decoding or manipulation on a webpage?

<figure><img src="../../.gitbook/assets/image (158).png" alt=""><figcaption><p>Found the profile directory</p></figcaption></figure>

Unfortunately FTK Imager does not ship with the ability to read sqlite files properly, so I had to extract the files to my host machine, then use DB Browser to open them up

<figure><img src="../../.gitbook/assets/image (159).png" alt=""><figcaption><p>How to export files from the image to your machine</p></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (161).png" alt=""><figcaption><p>Opening the file up in DB Browser for SQLite</p></figcaption></figure>

Navigating to the moz\_places table, we get a list of the user's past browsing history, which contained a suspicious looking URL

<figure><img src="../../.gitbook/assets/image (162).png" alt=""><figcaption></figcaption></figure>

Curling the AWS URL, we get an encoded string, which dCode's cipher identifier made out to be base64-encoded. From there, we can decode the string using base64 and obtain the flag
