### NORTHLAND-WKS01

Lets start with the machine with the highest number of logs, `NORTHLAND-WKS01`. Based on the name, we can assume this is an employee's workstation. Select the "+" symbol in the previous menu to filter all logs containing this agent name.

<figure><img src="../../.gitbook/assets/image (308).png" alt=""><figcaption></figcaption></figure>

To sort by time, just select the "Time" field to sort in descending (older -> newer) or ascending (newer -> older, the default option). Looking at the available fields, we can see that we're likely working with Windows logs, which can contain a slew of information, both valuable and junk.&#x20;

<figure><img src="../../.gitbook/assets/image (309).png" alt=""><figcaption></figcaption></figure>

We'll start by looking through `data.win.system.eventID`, which can give us clues to what system logs are enabled on the system. Looking through the top values, we can see some IDs within the range of 1-11, which indicate the presence of [Sysmon](https://learn.microsoft.com/en-us/sysinternals/downloads/sysmon).&#x20;

<figure><img src="../../.gitbook/assets/image (310).png" alt=""><figcaption></figcaption></figure>

<details>

<summary>Sysmon Event IDs</summary>

The following is a summarised list of the available Event IDs in Sysmon

* 1 - Process creation
* 2 - A process changed a file creation time
* 3 - Network connection
* 4 - Sysmon service state stopped
* 5 - Process terminated
* 6 - Driver loaded
  * Potential flag for BYOD (bring your own driver) attacks, but less common
* 7 - Image loaded
  * Tracks loading of DLLs (contrary to what the name suggests)
* 8 - CreateRemoteThread
  * Can aid in identifying injected threats, but may be generated by legitimate programs
* 10 - ProcessAccess
  * Useful for spotting remote code injection and memory dumping
* 11 - FileCreate
  * May be challenging to identify malicious activity in here, since file creations occur frequently
  * Can be helpful in correlating/identifying a file’s origins
* 12 - RegistryEvent (Object create and delete)
* 13 - RegistryEvent (Value set)
  * Both 12 and 13 are useful in looking out for malicious registry edits (e.g. to add persistence through Run keys)
* 15 - FileCreateStreamHash
  * Relates to file streams (e.g. ADS) and “mark of the web” pertaining to external downloads
* 16 - Sysmon config state changed
  * Monitors and logs alterations to sysmon configuration
  * Useful for identifying tampering of service
* 17 - Pipe created
* 18 - Pipe connected
  * Records pipe creations and connections (inter-process communications)
  * Can be used to observe malware’s inter-process communications, usage of PsExec and SMB lateral movement
* 22 - DNSEvent
* 23 - FileDelete
* 25 - ProcessTampering
  * Alerts on suspicious behaviors (e.g. process herpadering)

</details>

<figure><img src="../../.gitbook/assets/image (312).png" alt=""><figcaption></figcaption></figure>

On its own, not very suspicious, but since all we have left are system events we might as well investigate them just to check. These event logs gives us a username that we can use for future searches (`c.shaw`), which we can use now.

<figure><img src="../../.gitbook/assets/image (313).png" alt=""><figcaption></figcaption></figure>

Some new event IDs have popped up, most notably Event ID 3, indicating network connections. We can investigate them later, first we should see what the `installer.exe` we saw earlier was.

Search for \*installer.exe\*, and we can see some unusual processes

<figure><img src="../../.gitbook/assets/image (314).png" alt=""><figcaption></figcaption></figure>

The installer program seems to be spawning `cmd.exe`, the Windows command prompt. Very unusual, so we can investigate further. We get a new process ID, 7128, which we can filter again.

<figure><img src="../../.gitbook/assets/image (315).png" alt=""><figcaption><p>Nothing interesting unfortunately, lets go back</p></figcaption></figure>

Unfortunately, we don't really see any logged commands that the program was executing. Nevertheless, we can log this in our notepads (or brains) as a potential reverse shell, and move on. We can also note when this file was created, by using events with Event ID 11 and looking at `data.win.system.systemTime` field.&#x20;

<figure><img src="../../.gitbook/assets/image (316).png" alt=""><figcaption></figcaption></figure>

With a user directory provided, we can search for events related to files contained in this directory (note: you may need to alter your queries, for me I found that `*Users*` works fine to search for files in the user directories)

<figure><img src="../../.gitbook/assets/image (317).png" alt=""><figcaption></figcaption></figure>

Scrolling down, we start seeing some very suspicious event logs

<figure><img src="../../.gitbook/assets/image (318).png" alt=""><figcaption></figcaption></figure>

Two new files, `print.exe` and `updater.exe` . Based on the timestamps of the event logs so far, we can assume that the three files we've found so far are malicious, and perform different functions

* `installer.exe` - The initial payload, when ran will give the attacker a foothold on the system
* `print.exe` - Either a persistence mechanism or a privilege escalation exploit
* `updater.exe` - Possibly a second reverse shell

We can follow this trail by searching for the programs' event IDs, and continuing until we hit an unusual process.

<figure><img src="../../.gitbook/assets/image (320).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (321).png" alt=""><figcaption></figcaption></figure>

<figure><img src="../../.gitbook/assets/image (322).png" alt=""><figcaption></figcaption></figure>

We stumble across a 4th binary, `dump.exe`. We don't know what it does, so lets search up the binary's name and look for events generated.

<figure><img src="../../.gitbook/assets/image (323).png" alt=""><figcaption><p>Oh dear thats not very good</p></figcaption></figure>

As shown in the screenshot, the 3rd and 4th event logs are call traces, generated by Sysmon with an Event ID of 10 (ProcessAccess). This [article](https://www.ultimatewindowssecurity.com/securitylog/encyclopedia/event.aspx?eventid=90010) notes the properties of Event ID 10

> The process accessed event reports when a process opens another process, an operation that’s often followed by information queries or reading and writing the address space of the target process. This enables detection of hacking tools that read the memory contents of processes like Local Security Authority (Lsass.exe) in order to steal credentials for use in Pass-the-Hash attacks. Enabling it can generate significant amounts of logging if there are diagnostic utilities active that repeatedly open processes to query their state, so it generally should only be done so with filters that remove expected accesses.

Looks like we found the smoking gun. We can almost certain confirm that the 4 binaries found so far are all malicious, with the latest `dump.exe` being a credentials dumping utility used to read the memory of the LSA process and steal credentials stored in it. Finally, we can search for events with Event ID 3 (NetworkConnection) to see if the attacker attempted to pivot to a different host

<figure><img src="../../.gitbook/assets/image (324).png" alt=""><figcaption></figcaption></figure>

`scanner.exe` is likely a network mapper, like Nmap. We also see connections to WinRM (port 5985) as shown here, around 16 minutes later. Unfortunately, there isn't enough information to correlate this connection with the previous credentials dumping process, although it does reveal a potential attacker IP address (172.16.100.156). We will take note of this and return back here later.

<figure><img src="../../.gitbook/assets/image (325).png" alt=""><figcaption></figcaption></figure>

### NORTHLAND-HR01

With the previously-discovered information, we can narrow our search again, to all events past 1003. Lets also search for any network connections made, which can indicate lateral movement

<figure><img src="../../.gitbook/assets/image (326).png" alt=""><figcaption></figcaption></figure>

Looks like the attacker was able to pivot successfully. Since the destination port is using WinRM, we can search for potential Pass-the-Hash attacks, using the following set of criteria

* Event ID 4624&#x20;
* Logon Process Name NtLmSsp
* Authentication Package NTLM

## Creator's Notes

As of time of writing, Secure The Network v3 is the most complex challenge I've written so far, using new techniques for investigations that I've never used before. As such, the challenge was labelled as an experimental challenge, and participants were encouraged to open tickets for help. This was also originally meant to be part 2 of Secure The Network v2, but due to troubleshooting issues and time was later broken up into independent challenges.

Based on feedback provided by participants, Secure The Network was a very fun and unique challenge due to the use of an SIEM, which is not commonly seen in CTFs. However, due to the unique use of SIEMs, many participants also encountered issues with trying to search for logs, which may have been a contributing factor to the low number of solves for this challenge.

The original question bank was intended to have more questions, including discovering the user used for a DCSync attack which allowed the attacker to gain access to the DC. However, due to misconfigurations with the audit policies, many events were not captured by the SIEM, and led to several holes in the final reconstructed attack chain. Nevertheless, I will deem this proof of concept a success, as it seemed to be successful in counteracting AI tools like Codex and Antigravity, which are used commonly to auto-solve challenges.
